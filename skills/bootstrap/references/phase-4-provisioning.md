# Phase 4 — Code + tracking provisioning

After scaffold and backlog are ready, present the summary:

> **[Project Name] — Scaffold Complete**
>
> **Files generated:** [count] files across [domains]
> **Database tables:** [list]
> **Auth plugins:** [list]
> **BullMQ queues:** [list]
> **Test stubs:** [count] unit, [count] integration, [count] E2E
>
> **Backlog:** [N] epics, [N] stories, [N] total story points
> **MVP scope:** [N] stories, [N] story points

**In interactive mode (default):** ask about each provision step **one at a time**, waiting for an explicit yes/no before proceeding to the next. Do NOT present them as a batch.

**In auto-accept mode:** run each step sequentially without asking. Show what you're doing (so the builder can follow along or interrupt) but don't wait for approval. If any step fails (auth error, tool unavailable, etc.), stop and report — don't skip.

Ask/run in this order:

### Pre-flight checks

Before asking about each step, verify tool availability:
```bash
gh auth status          # GitHub CLI authenticated?
doctl account get       # DigitalOcean CLI authenticated?
# Asana MCP — attempt to list workspaces via MCP tools
# Shortcut MCP — attempt to list projects via MCP tools
# Linear MCP — attempt to list teams via MCP tools
```

If a tool is not available for a given step, say so and offer to skip with manual instructions instead.

### Step 1: GitHub repository

Ask:
> **Would you like me to set up a GitHub repository?**
> I can either:
> - **Create a new repo** — tell me which org (e.g., `fractionwork`, your personal account, or another org)
> - **Use an existing empty repo** — give me the URL (e.g., `git@github.com:myorg/my-project.git`)
>
> *(Requires: `gh` CLI authenticated)*

**If creating a new repo:** Ask which org/owner to create it under. Do NOT assume `fractionwork` — ask explicitly. Then:
```bash
gh repo create [owner]/[project-name] \
  --private \
  --description "[one-liner]" \
  --source . \
  --remote origin

git add -A
git commit -m "feat: bootstrap [project-name] from devhawk-seed

Scaffold generated by DevHawk Bootstrap.

Stack: Next.js 16 · Drizzle · Better Auth · BullMQ · shadcn/ui
Infrastructure: DO App Platform · Managed PG 16 · Managed Valkey (Redis-compatible)

Epics: [N] ([N] MVP)
Story points: [N] total ([N] MVP)"

git branch -M main
git push -u origin main
```

**If using an existing repo:** Verify it's empty (no commits), then:
```bash
git remote add origin [repo-url]

git add -A
git commit -m "feat: bootstrap [project-name] from devhawk-seed

Scaffold generated by DevHawk Bootstrap.

Stack: Next.js 16 · Drizzle · Better Auth · BullMQ · shadcn/ui
Infrastructure: DO App Platform · Managed PG 16 · Managed Valkey (Redis-compatible)

Epics: [N] ([N] MVP)
Story points: [N] total ([N] MVP)"

git branch -M main
git push -u origin main
```

If the existing repo has commits, STOP and ask the user how to proceed. Do NOT force-push.

If no or `gh` unavailable: tell the user to create the repo manually and provide the git commands. The project pulls future seed updates on demand via the `update-seed` skill (Tier-2 substrate) and the user-level installer (Tier-1 skills) — see `docs/seed-distribution.md`. There is no auto-sync topic/marker to set up.

### Step 2: Project management backlog

Detect which project management MCP servers are available — supported: **Asana**, **Shortcut**, **Linear**. Then ask:

**If multiple are connected:**
> **Where would you like me to create the project backlog?**
> - **Asana** — I'll create a project with sections and populate epics/stories
> - **Shortcut** — I'll create a project with workflow states and populate epics/stories
> - **Linear** — I'll create a team/project and populate epics/issues
> - **Skip** — I'll save the backlog to `backlog.md`

**If exactly one is connected:**
> **Would you like me to create a [PM tool] project and populate the backlog?**
> This will create [N] epics and [N] stories in your [PM tool] workspace.

**If none are connected:**
> **No project management MCP is connected (Asana, Shortcut, or Linear).**
> I'll save the backlog to `backlog.md`. You can populate your PM tool later.

#### Asana backlog creation
Use Asana MCP for task creation + `scripts/asana_ops.py` (REST) for operations MCP can't do. Follow `docs/asana-best-practices.md`.

1. **Create project** via MCP — Name: "[Project Name]", notes include one-liner + repo link + stack + team
2. **Attach 6 standard fields** via REST (MCP can't do this):
   ```bash
   # Fraction Priority, Task Type, Story Points, Release, Sprint, Task Progress
   python3 scripts/asana_ops.py  # or inline REST calls
   # Or just run hygiene immediately after project creation — it auto-attaches all 6
   python3 scripts/asana_ops.py --hygiene <PROJECT_GID>
   ```
   Field GIDs: see `docs/asana-best-practices.md`
3. **Create standard sections** via REST — INBOX (top), BACKLOG, TODO, WIP, Ready for Review, Ready for Testing, Ready for Release, DONE (in order). INBOX captures pre-stakeholder ideas; see `docs/asana-best-practices.md` for the per-section hygiene table.
4. **Add required admins** — Jeremy King + Alyssia Maluda as project Admins
5. **Create tasks for each epic** in TODO section — set Fraction Task Type = EPIC, Release = "Phase 1"
6. **Create subtasks for each story** — set Task Type = Story, Story Points, Fraction Priority, Release = "Phase 1"
7. **Set project metadata** via REST — start_on, due_on from discovery
8. **Post initial status update** via REST
9. **Add to portfolio** (if on Advanced plan) or index project (if Starter)
10. **Mark E0 complete**

#### Shortcut backlog creation
Use the Shortcut MCP for reads and `scripts/shortcut_ops.py` (REST) for setup that MCP can't do. Follow `docs/shortcut-best-practices.md`.

1. **Create project** via MCP — Name: "[Project Name]", description includes one-liner + repo link + stack
2. **Ensure workspace-level workflow** has the standard states: Inbox, Unscheduled, Ready for Development, In Development, Ready for Review, Ready for Deploy, Completed (Inbox first if missing — see best-practices doc)
3. **Ensure Priority custom field** exists with P0–P3 values: `python3 scripts/shortcut_ops.py --setup`
4. **Create epics** for each epic in the backlog
5. **Create stories** under each epic with description, acceptance criteria, `estimate`, `story_type`, Priority, parent epic. New stories land in **Unscheduled** (BACKLOG); stakeholder-pending ideas land in **Inbox**.
6. **Mark E0 stories as Completed**

#### Linear backlog creation
Use the Linear MCP. Linear's data model is simpler — no per-project sections, just teams + projects + issues + cycles.

1. **Create or pick a team** — confirm with the user which team owns the work
2. **Create a project** under the team — Name: "[Project Name]", description includes one-liner + repo link + stack
3. **Create epics** as parent issues (Linear models epics as issues with sub-issues)
4. **Create stories** as sub-issues under each epic with description, acceptance criteria, estimate, priority, labels
5. **Mark E0 stories as Done**

#### Fallback
If the user skips or MCP is unavailable, write the backlog to `backlog.md` using the format defined in `docs/backlog-format.md`:

```markdown
# [Project Name] Backlog

## E1: [Epic Name] [P0]
> [one-line scope]

### [E1-S1] [Story title] (N SP) @unassigned — TODO
- [ ] [acceptance criterion]
- [ ] [acceptance criterion]

## v2 (deferred)
- [deferred item]
```

Each `### [EX-SY]` heading is a work unit with story points, assignee, and status. The `next-task` skill parses this format to pick the next card when no PM is connected.

> Backlog saved to `backlog.md`. To populate a PM tool later, connect an MCP server and say **"provision backlog"**.

## Local verification gate *(between Phase 4 and Phase 5)*

**Cloud provisioning does NOT run before this gate passes.** The single most common cause of "deployed but broken" first deploys is provisioning DO clusters before the app has been exercised locally. A schema bug, a missing env var, an auth misconfiguration, or a queue wiring mistake costs 30 seconds to fix locally and 30 minutes to diagnose against a managed cluster — plus pollutes the live DB with a broken migration that has to be cleaned up.

State this verbatim to the builder:

> **Before we provision DO, the app must be running locally with the golden path verified.**
>
> What "verified" means:
> 1. `pnpm dev` starts cleanly — web + worker + PG + Redis + MinIO all healthy
> 2. `pnpm db:migrate` applies cleanly against the local DB
> 3. You can sign up at `http://localhost:3000/sign-up`, see the dashboard, and sign out
> 4. At least one core domain mutation (the main thing your app does) succeeds end-to-end
> 5. `pnpm typecheck` and `pnpm lint` are green
> 6. If your app uses object storage: a test upload to local MinIO works
>
> Why this gate exists: bugs caught here cost seconds. Bugs caught after `doctl apps create` cost minutes-to-hours and may leave the prod DB in a broken state. **No exceptions** — even if you're confident, run the gate.

Then AskUserQuestion:
- header: "Local verification" — Verified, proceed to cloud / Need time, pause bootstrap / Skip gate (NOT recommended)

**Verified, proceed to cloud** → continue to Phase 5.

**Need time, pause bootstrap** → save state and exit cleanly. The builder runs `pnpm dev`, exercises the app, fixes anything broken, then re-invokes bootstrap (or says "resume bootstrap") to enter Phase 5 directly. State the resume command explicitly.

**Skip gate** → Only honor this if the builder explicitly says they accept the risk (e.g., "skip, I know what I'm doing" or auto-accept mode + `--skip-verify`). Even then, log a warning in the final Phase 5 report that the gate was bypassed.

In **auto-accept mode**, the gate is still a hard pause unless `--skip-verify` was passed. Auto-accept covers per-step approval; it does NOT bypass safety gates by default.
